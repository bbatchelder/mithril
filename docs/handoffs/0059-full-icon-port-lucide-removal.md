# 0059 — Full icon port + lucide-react removal

- **Date:** 2026-05-26
- **Focus:** Replace the hand-curated 25-icon subset with the complete Blueprint icon set, and drop the lone third-party icon dependency (`lucide-react`).
- **Branch / commit:** `icons-full-port` (off `main` @ bea2355) — uncommitted at time of writing.

## Summary

Vendored the **full @blueprintjs/icons v6.15 set — 706 glyphs** (both 16px and 20px grids) into
`src/components/ui/icons/index.ts`, generated by a new reproducible script `tools/gen-icons.mjs`.
Removed `lucide-react` entirely: the `Button` loading spinner now uses our own `Spinner`, and the
`src/App.tsx` showcase icons now use our `Icon`. The public icon API is unchanged — same `ICON_GLYPHS`,
`IconGlyph`, and `IconName` exports — so every existing consumer keeps working; `IconName` is now a
706-member string-literal union.

## Decisions made (and why)

- **Single static `ICON_GLYPHS` map, NOT per-icon modules.** Matches the existing structure and the
  synchronous `icon="name"` string API. Same model as Blueprint's `allPaths` "static" mode.
- **`IconName` emitted as an explicit string-literal union + `ICON_GLYPHS: Record<IconName, IconGlyph>`
  with NO `as const`.** Declaring the 706-entry object `as const satisfies …` makes `ICON_GLYPHS[icon][grid]`
  blow up with **TS2590 "union type too complex"** (it materializes a union of ~1,412 literal path tuples).
  The explicit-union + total-`Record` form keeps type-safe names and exhaustiveness (missing glyph = compile
  error) without TS inferring literal tuples for any path. **Do not reintroduce `as const`/`keyof typeof`
  on the data object** — that's what triggers TS2590.
- **Button loading spinner uses `[&_path:last-child]:!stroke-current`** so the spinner head inherits the
  button's text color (faithful to Blueprint), while the track stays faint.

## Gotchas / things to know

- **No tree-shaking — this is an accepted trade-off, by decision.** The single-object + runtime
  `ICON_GLYPHS[icon]` lookup means importing `Icon` bundles **all 706 glyphs** (~591 KB source; verified all
  706 grid arrays survive into the production bundle even though the demo app renders ~27 distinct icons).
  Tree-shaking operates on module bindings, not object properties, and the lookup is dynamic — so the whole
  map is retained. The TS typing (union vs `as const`) is irrelevant to this; types are erased before bundling.
  - **Why we accept it:** shadcn-style "you own the source" — a consumer who cares about bytes trims unused
    entries from their copied `index.ts` (the `Record` totality + the union keep that type-safe). The project
    isn't packaged for npm distribution yet.
  - **If that changes:** the real fix is per-icon modules with a value API
    (`import { add } from ".../icons"; <Icon icon={add} />`, the lucide/heroicons model) — tree-shakeable but
    drops the `icon="name"` string ergonomics and the central autocomplete union. A lazy `import()` loader
    (Blueprint's dynamic mode) is the other option but makes rendering async.
- **Regenerating icons:** `node tools/gen-icons.mjs`. It reads the per-icon path files from
  `tools/blueprint-reference/node_modules/@blueprintjs/icons/lib/esm/generated/{16,20}px/paths/*.js`
  (the `paths/index.js` barrel uses extensionless re-exports that Node ESM rejects, so the script imports the
  individual files directly). Bump `@blueprintjs/icons` in `tools/blueprint-reference` and re-run to pick up
  new icons.
- `lucide-react` is gone from `package.json` and the lockfile; no source references remain.
