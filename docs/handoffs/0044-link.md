## API

```tsx
// Default — primary color, always underlined
<Link href="#">Blueprint Link</Link>

// Underline variants
<Link href="#" underline="always">Always underlined (default)</Link>
<Link href="#" underline="hover">Underline on hover only</Link>
<Link href="#" underline="none">No underline</Link>

// Color / intent variants
<Link href="#" color="primary">Primary (default)</Link>
<Link href="#" color="success">Success</Link>
<Link href="#" color="warning">Warning</Link>
<Link href="#" color="danger">Danger</Link>
<Link href="#" color="inherit">Inherits surrounding text color</Link>

// Inline in body text
<p>
  Visit the <Link href="#">Blueprint documentation</Link> for more details.
</p>
```

## Design decisions

- **Blueprint DOES have a Link component**: Contrary to the task description's assumption, Blueprint
  v6.15 ships a `Link` component in `packages/core/src/components/link/`. It was added recently (2025).
  We port its exact API: `underline` and `color` props with the same default values (`underline="always"`,
  `color="primary"`). The blueprint-reference gallery uses this component directly.

- **Direct palette colors instead of `--intent-*-text` tokens**: Blueprint's Link uses
  `$pt-dark-intent-text-colors` (blue5, green5, orange5, red5) for dark-mode intent colors — the raw
  palette values. Our `--intent-*-text` semantic tokens in dark mode use `color-mix()` formulas (designed
  for Button/Tag contexts, which needed slightly different mixed values). Using those would produce
  mismatches (e.g. blue-3 51% white → rgb(149,184,235) vs blue-5 #8abbff → rgb(138,187,255)). Fix:
  use `text-blue-2 dark:text-blue-5`, `text-green-2 dark:text-green-5`, etc. — literal palette classes.

- **Focus ring**: Blueprint's `_link.scss` does not define a focus style — it inherits from the browser
  default or Blueprint's global focus ring (which uses the `outline` property via `pt-focus-indicator-color`).
  Our Tailwind v4 setup doesn't globally override focus for anchors. We omit an explicit focus style for
  now (same as Blueprint's component definition). A future pass could add `focus-visible:outline-ring`.

- **Global anchor styles vs component styles**: Blueprint's `_typography.scss` defines global `a { … }`
  rules (text-decoration: none at rest, underline on hover, link-color). The Link component overrides these
  via `.bp6-link-underline-*` classes. Our Link component's CVA classes take precedence over any global
  anchor styles that may exist.

- **`link-hover-nope` key**: The gallery has a deliberate extra specimen (`link-hover-nope` — the "always"
  underline variant row) that doesn't have a matching `data-compare` key on the Blueprint side. The harness
  reports it as "only in analyst" (greyed out informational line) — not a failure. This is intentional to
  show the always-underlined state without polluting the compare set with a duplicate of `link-default`.
