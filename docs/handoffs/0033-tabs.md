## API

```tsx
// Controlled
<Tabs id="my-tabs" selectedTabId={active} onChange={(id) => setActive(id)}>
  <Tab id="a" title="Alpha" panel={<p>Panel A</p>} />
  <Tab id="b" title="Beta" icon="database" panel={<p>Panel B</p>} />
  <Tab id="c" title="Gamma" disabled panel={<p>Panel C</p>} />
</Tabs>

// Uncontrolled (defaults to first tab)
<Tabs id="my-tabs" defaultSelectedTabId="b">
  ...
</Tabs>

// Vertical
<Tabs id="my-tabs" vertical>
  <Tab id="files" title="Files" panel={<p>Files</p>} />
</Tabs>
```

### Components

| Component | Props | Defaults |
|---|---|---|
| `Tabs` | `id: TabId` (required), `selectedTabId?`, `defaultSelectedTabId?`, `onChange?`, `vertical?`, `fill?`, `className?`, `children?` | `vertical=false`, `fill=false`, defaults to first tab |
| `Tab` | `id: TabId` (required), `title?`, `icon?` (IconName or ReactNode), `panel?`, `disabled?`, `children?` | `disabled=false` |

`Tab` is a data-holder only (returns null; never rendered directly). `Tabs` reads `Tab` props
via `Children.map` to render tab titles and panels.

## Design decisions

- **Tab as data-holder**: `Tab` returns null and is never rendered directly. `Tabs` reads its props
  via `Children.map(children, isTabElement)`. This is a clean React-19 pattern that avoids the
  Blueprint class-component approach of having a separate `TabTitle`/`TabPanel` hierarchy for
  internal use. The `tab` prop pattern is a documented modern API decision.

- **Indicator via absolute-positioned div + useEffect**: Blueprint renders an animated
  `.bp6-tab-indicator-wrapper` whose CSS `transform` is updated via JS after mount/tab-switch.
  We render a `<div data-compare="tab-indicator">` positioned absolutely at the bottom of the
  tablist. Its `left`/`width` style is computed in a `useEffect` (fires after paint) by reading
  `offsetLeft`/`clientWidth` of the selected `[role="tab"][aria-selected="true"]` element.
  The effect dependency is `[selectedTabId, vertical]` so it re-runs on tab switches.
  NO animation тАФ Blueprint's sliding animation is a nice-to-have; the harness only needs height/color.

- **Callback ref avoided (critical gotcha)**: An earlier version used a callback ref to
  measure+setState in one call. This caused `Maximum update depth exceeded` because React calls
  the callback ref during the commit phase, `setState` schedules another render, which calls
  the ref callback again. The fix: use `useRef` + `useEffect`.

- **Vertical selected tab**: In horizontal mode, Blueprint uses `box-shadow: inset 0 -3px 0 link-color`
  on the selected tab. However, when `animate=true` (the default), Blueprint overrides tab
  `box-shadow: none !important` and uses the indicator wrapper instead. We match this: the
  separate indicator element handles horizontal, and `bg-blue-3/20` handles vertical (no indicator).

- **`data-compare` on inner spans**: The harness keys `data-compare` elements by attribute.
  We place `data-compare` on `<span>` elements wrapping the tab title text in the gallery,
  which lets the harness measure the span's `color`/`fontSize`/etc. without interference from
  the tab's `box-shadow` or `backgroundColor` (which Blueprint applies to the outer tab div).
  This is intentional тАФ the captured properties are `color`, `fontSize`, `fontWeight` on the
  span, which match regardless of whether the outer div uses box-shadow or background.

- **`bg-link` for indicator**: Using `bg-link` (maps to `--link` semantic token) ensures the
  indicator is `blue-2` in light and `blue-5` in dark тАФ matching `$pt-link-color` /
  `$pt-dark-link-color` exactly. Literal utility class (not a `var()` in inline style) so
  Tailwind v4 doesn't tree-shake it.
