## The playground framework (`src/playground.tsx`)

`PLAYGROUNDS: Record<componentId, PlaygroundConfig>`. A component with an entry shows a
Playground section; everything else falls back to its Examples gallery. A config:

```ts
interface PlaygroundConfig {
  initial: Record<string, unknown>;          // starting prop values (also the "Custom…" reset)
  controls: Control[];                        // one row per control
  presets?: { name: string; props: {...} }[]; // named prop bundles in a dropdown
  render: (props) => React.ReactNode;         // the live instance — OWNS the JSX
  code:   (props) => string;                  // the live code snippet (can be multi-line)
}
```

`Control` kinds: `boolean` (→ `Switch`), `enum` (→ `SegmentedControl` ≤3 options else
`HTMLSelect`; override with `widget: "segmented" | "select"`), `text` (→ `InputGroup`),
`number` (→ `NumericInput`, needs `min`/`max`/`stepSize`). Shared option sets live at the top:
`INTENTS`, `VARIANTS`, `SIZES`, `ALIGN`, `ICONS`/`ICONS_REQ`, `ELEVATIONS`, `ENTITY_SIZES`,
`UNDERLINE`, `LINK_COLORS`, `LAYOUTS`, `BUTTON_POSITIONS`.

Helpers: `jsx(tag, attrs, children?)` builds a code string, omitting empty/false/undefined attrs
(use the `"{expr}"` escape hatch for non‑string expressions like `size={SpinnerSize.LARGE}`).

### Recipe — adding a config

1. Pick `initial` values; add a `controls` row per prop you want to expose.
2. Write `render` — it returns the live JSX. **It owns structure**, so compound components just
   build their own children/items from scalar props (see `menu`, `tabs`, `breadcrumbs`).
3. Write `code` to mirror the live props (use `jsx()` or a hand‑built multi‑line string).
4. Optionally add `presets`.
5. `pnpm build` + eyeball at `#showcase/<id>` in **both** themes.

### Gotchas (learned the hard way)

- **Inline controls + inline demo components only** — everything in batches 1–3 is non‑portaled,
  so dark mode works via the ancestor with **no `dark` threading**. The moment you add an
  overlay (Tooltip/Popover/Dialog/Drawer/Toast/ContextMenu/Omnibar) you must thread `dark`
  (see the overlays section below).
- **Controlled components need a stateful wrapper.** `render` is a pure function of props, so a
  component that must be controlled (RadioGroup) can't hold selection itself. Define a small
  stateful component (see `RadioGroupDemo`) and return it from `render`. Same pattern will be
  needed for Select/Suggest/MultiSelect/TagInput/date pickers.
- **`render` receives `props` typed as `Record<string, never>`** (a deliberate simplification so
  configs read cleanly). Accessing `p.anything` yields `never`, which is assignable anywhere —
  so `p.foo || undefined`, `disabled={p.foo}`, ternaries, etc. all typecheck. Don't fight it.
- **Vite HMR leaves stale playground state** when you navigate to a *newly added* config (empty
  stage / unseeded `initial`). It is a **dev‑only artifact** — a hard reload (ignore cache)
  fixes it every time and the production build is unaffected. Don't chase it as a bug.
- **`defaultIsOpen`/`defaultChecked`/`defaultValue` only apply on mount.** If a control toggles
  something that maps to a `default*` prop (e.g. Section's open state, EditableText multiline),
  add a `key` that changes with that control so the instance remounts (see `section`,
  `editable-text`).
- **Icon‑name controls must use names in `ICONS`/`ICONS_REQ`.** A preset that sets an icon not in
  the option list renders a `<select>` with no matching option. Add the glyph to the shared list
  (that's why `caret-down` is there).

## Coverage

**Configured (30):** button, tag, callout, spinner, progress-bar, switch, checkbox, input-group,
icon, slider, anchor-button, text-area, html-select, numeric-input, segmented-control,
editable-text, card, entity-title, non-ideal-state, link, divider, collapse, menu, tabs,
breadcrumbs, html-table, card-list, section, control-card, radio.

**Remaining (33),** grouped by approach:

### A. Easy — scalar controls, no framework change (8)
`text`, `skeleton`, `button-group`, `file-input`, `form-group`, `control-group`, `hotkeys`
(KeyCombo), `navbar`. Same pattern as batches 1–3: scalar controls, `render` builds any structure.
`button-group`/`control-group`/`form-group`/`navbar` wrap a couple of sample children. **Do these
first — fastest wins.** (`navbar` may be borderline‑low‑value; gallery‑only is an OK call.)

### B. Overlays — need a small framework extension (9)
`tooltip`, `popover`, `dialog`, `drawer`, `alert`, `multistep-dialog`, `toast`, `context-menu`,
`omnibar`. These portal to `document.body`, so the playground stage must:
  - render a **trigger** (a Button) that opens the overlay, and
  - **thread `dark`** into the overlay (`dark={dark}` from `useDark()` / `DarkContext`), since the
    portal renders outside the app's `.dark` ancestor — this is the #1 overlays gotcha in the
    [overlays recipe](../.claude/skills/mithril/reference/overlays.md).
  Suggested extension: let a config opt into receiving `dark` (e.g. `render: (props, ctx) =>`
  where `ctx.dark` comes from `useDark()`), or add an `overlay: true` flag so `Playground`
  supplies a managed `open`/`onOpenChange` + `dark`. Modal ones (Dialog/Drawer/Alert) lock body
  scroll — fine for a triggered open, but don't render them open‑by‑default in the stage.
  Note `MenuPopover` for menu‑in‑popover, and the `anchorOnly` / dismiss‑race notes in the recipe.

### C. Stateful / heavier inputs — bespoke stateful wrappers (13)
`select`, `suggest`, `multi-select`, `tag-input`, `time-picker`, `date-picker`, `date-input`,
`date-range-picker`, `date-range-input`, `timezone-select`, `data-table`, `tree`, `panel-stack`.
Each needs a small stateful demo wrapper (like `RadioGroupDemo`) holding value/selection, plus
sample data (options, columns+rows, tree nodes). The Select/Suggest/MultiSelect family **also
portals**, so combine with the overlay handling from group B. See the
[forms](../.claude/skills/mithril/reference/forms.md), [data](../.claude/skills/mithril/reference/data.md),
and [dates](../.claude/skills/mithril/reference/dates.md) recipes.

### D. Infrastructure — playground may not fit (~3)
`resize-sensor`, `overflow-list`, `portal`. Behavioral/layout utilities; a controls‑driven single
instance may be low value. Consider leaving them on their Examples gallery, or build a bespoke
demo. Decide per‑component; don't force a config — a gallery‑only page is a fine outcome.

## Conventions & verification

- **Branch off `main`, build, verify, open a PR, let CI (`verify`) pass, merge.** One batch per
  PR has worked well. (Don't commit straight to `main` — that happened once with batch 2.)
- **Verify before merge:** `pnpm build` (typecheck + vite) and `pnpm test` (344 currently) green.
  Then eyeball new playgrounds at `#showcase/<id>` in **both** themes — and **hard‑reload** after
  adding configs because of the HMR staleness above. Console should be clean
  (`?theme=dark#showcase/<id>` for the dark pass).
- **Suggested next step:** knock out group **A** (one quick PR), then design the group **B**
  overlay extension (one PR for the framework change + a first overlay or two), then chip away at
  **C** a few at a time.
