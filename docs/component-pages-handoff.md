## The playground framework (`src/playground.tsx`)

`PLAYGROUNDS: Record<componentId, PlaygroundConfig>`. A component with an entry shows a
Playground section; everything else falls back to its Examples gallery. A config:

```ts
interface PlaygroundConfig {
  initial: Record<string, unknown>;          // starting prop values (also the "Custom‚Ä¶" reset)
  controls: Control[];                        // one row per control (omit the panel entirely if [] and no presets)
  presets?: { name: string; props: {...} }[]; // named prop bundles in a dropdown
  render: (props, ctx) => React.ReactNode;    // the live instance ‚Äî OWNS the JSX. ctx is for overlays (see below)
  code:   (props) => string;                  // the live code snippet (can be multi-line)
}
```

**Overlay context (`ctx`, batch 5).** `render`'s second arg is
`{ dark, open, setOpen }` ‚Äî supplied by `Playground` for **overlay** configs (inline configs
ignore it). `dark` comes from `useDark()` and must be threaded into every portaled overlay
(`dark={ctx.dark}`). `open`/`setOpen` is a managed open state: overlays that render only a
portaled panel (Dialog/Drawer/Alert/MultistepDialog/Omnibar) get a stage trigger
`<Button onClick={() => ctx.setOpen(true)}>` plus `open={ctx.open} onOpenChange={ctx.setOpen}`.
Tooltip/Popover/ContextMenu self-manage open (hover/click/right-click) and only need `ctx.dark`.
Toast is imperative ‚Äî see `ToasterDemo`/`ToasterButton` (a `<Toaster dark>` wrapper + `useToaster()`).
`Playground` is mounted with `key={id}` in `App.tsx` so this `open` state (and all prop state)
resets per component.

`Control` kinds: `boolean` (‚Üí `Switch`), `enum` (‚Üí `SegmentedControl` ‚â§3 options else
`HTMLSelect`; override with `widget: "segmented" | "select"`), `text` (‚Üí `InputGroup`),
`number` (‚Üí `NumericInput`, needs `min`/`max`/`stepSize`). Shared option sets live at the top:
`INTENTS`, `VARIANTS`, `SIZES`, `ALIGN`, `ICONS`/`ICONS_REQ`, `ELEVATIONS`, `ENTITY_SIZES`,
`UNDERLINE`, `LINK_COLORS`, `LAYOUTS`, `BUTTON_POSITIONS`.

Helpers: `jsx(tag, attrs, children?)` builds a code string, omitting empty/false/undefined attrs
(use the `"{expr}"` escape hatch for non‚Äëstring expressions like `size={SpinnerSize.LARGE}`).

### Recipe ‚Äî adding a config

1. Pick `initial` values; add a `controls` row per prop you want to expose.
2. Write `render` ‚Äî it returns the live JSX. **It owns structure**, so compound components just
   build their own children/items from scalar props (see `menu`, `tabs`, `breadcrumbs`).
3. Write `code` to mirror the live props (use `jsx()` or a hand‚Äëbuilt multi‚Äëline string).
4. Optionally add `presets`.
5. `pnpm build` + eyeball at `#showcase/<id>` in **both** themes.

### Gotchas (learned the hard way)

- **Inline (non‚Äëportaled) configs need no `dark` threading** ‚Äî dark works via the ancestor.
  **Overlays (batch 5) do** ‚Äî thread `ctx.dark` into every portaled overlay (the overlay context
  above). All 9 overlays are now configured; copy one as a template.
- **Controlled components need a stateful wrapper.** `render` is a pure function of props, so a
  component that must be controlled (RadioGroup) can't hold selection itself. Define a small
  stateful component (see `RadioGroupDemo`) and return it from `render`. Same pattern will be
  needed for Select/Suggest/MultiSelect/TagInput/date pickers.
- **`render` receives `props` typed as `Record<string, never>`** (a deliberate simplification so
  configs read cleanly). Accessing `p.anything` yields `never`, which is assignable anywhere ‚Äî
  so `p.foo || undefined`, `disabled={p.foo}`, ternaries, etc. all typecheck. Don't fight it.
- **Vite HMR leaves stale playground state** when you navigate to a *newly added* config (empty
  stage / unseeded `initial`). It is a **dev‚Äëonly artifact** ‚Äî a hard reload (ignore cache)
  fixes it every time and the production build is unaffected. Don't chase it as a bug.
- **`defaultIsOpen`/`defaultChecked`/`defaultValue`/`initialStepIndex` only apply on mount.** If a
  control toggles something that maps to such a prop (Section's open state, EditableText multiline,
  MultistepDialog's start step), add a `key` that changes with that control so the instance remounts
  (see `section`, `editable-text`, `multistep-dialog`).
- **Icon‚Äëname controls must use names in `ICONS`/`ICONS_REQ`.** A preset that sets an icon not in
  the option list renders a `<select>` with no matching option. Add the glyph to the shared list
  (that's why `caret-down` is there).

**Remaining (3),** grouped by approach:

### A. Easy ‚Äî scalar controls, no framework change ‚úÖ DONE (batch 4, PR #39)
`text`, `skeleton`, `button-group`, `file-input`, `form-group`, `control-group`, `hotkeys`
(KeyCombo), `navbar` ‚Äî all shipped. Same pattern as batches 1‚Äì3: scalar controls, `render` builds
any structure; the multi‚Äëchild wrappers (button-group/control-group/form-group/navbar) build their
own sample children. `navbar` got a heading/search/right‚Äëgroup toggle set rather than gallery‚Äëonly.

### B. Overlays ‚úÖ DONE (batch 5, PR #40)
`tooltip`, `popover`, `dialog`, `drawer`, `alert`, `multistep-dialog`, `toast`, `context-menu`,
`omnibar` ‚Äî all shipped. The framework gained the **overlay `ctx`** (`render(props, ctx)` with
`{ dark, open, setOpen }`; see "Overlay context" above) and `Playground` is now mounted with
`key={id}` so the managed `open` state resets per component. Modal ones open from a stage trigger
(not open‚Äëby‚Äëdefault). Toast uses an imperative `<Toaster>` + `useToaster()` wrapper. The only
gotcha hit: `useToaster()` needs a `<Toaster>` ancestor, **not** `<ToastProvider>`.

### C. Stateful / heavier inputs ‚Äî bespoke stateful wrappers ‚úÖ DONE
**DONE:** the **Select/Suggest/MultiSelect/TagInput family** (batch 6, PR #41),
**DataTable** (batch 7, PR #42), and the **dates/time + tree/panel-stack** set (batch 8).

- *Select family* ‚Äî each got a small stateful demo wrapper
  (`SelectDemo`/`SuggestDemo`/`MultiSelectDemo`/`TagInputDemo`) holding value/selection, with
  `ctx.dark` threaded into the three that portal (TagInput is inline, so it ignores `ctx`). They
  share `PG_FRUITS` + a `fruitMatch` predicate. No `key`/remount tricks ‚Äî selection lives in the
  wrapper's own state, not a `default*` prop.
- *DataTable* ‚Äî `DataTableDemo` holds the row data (so inline edits persist via `onCellEdit`);
  columns are `useMemo`'d on the `editable` toggle so the engine doesn't re-init sizing/order each
  render. Inline (no `ctx.dark`). Controls: selection mode, **row count** (6/50/1,000, sliced from a
  1,000-row `PG_TABLE_POOL`; the demo is keyed on count so changing it resets the slice), **height**
  (segmented `auto | 200 | 400 | 800` ‚Üí the `height` prop; a fixed height bounds the viewport so rows
  virtualize), **row height** (number ‚Üí `rowHeight` prop), gutter, resize/reorder, editable, loading.
  It also renders **contextual usage hints** under the grid (the selection gesture for the active
  mode, double-click-to-edit, drag-to-reorder). Two component facts worth knowing: (1) rows are
  **always** virtualized ‚Äî there's no `virtualized` prop; `height` just bounds the viewport so only
  the visible window renders (else it grows to fit). (2) DataTable has **no `fill`** ‚Äî columns are
  fixed-width and the grid is intrinsically sized to their sum + gutter, so a full-width wrapper would
  just leave empty space, not stretch columns (this is also [issue #43](https://github.com/bbatchelder/mithril/issues/43)).

Copy these as the template for any future stateful config.

- *Dates / time* (batch 8) ‚Äî each got a stateful demo wrapper holding its value/selection
  (`TimePickerDemo` / `DatePickerDemo` / `DateRangePickerDemo` / `DateInputDemo` /
  `DateRangeInputDemo` / `TimezoneSelectDemo`). Only the **input** variants portal a popover, so
  only `DateInput`/`DateRangeInput`/`TimezoneSelect` thread `ctx.dark` ‚Äî the inline calendars
  (`TimePicker`/`DatePicker`/`DateRangePicker`) inherit dark via the ancestor and ignore `ctx`
  (despite the earlier note here; go by which props expose a `dark` prop). Shared seed dates
  (`PG_DATE` = Jan 15 2026, `PG_TIME` = 2:30 PM, `PG_RANGE` = Jan 8‚Äì20) keep the live instances
  deterministic. No `key`/remount tricks ‚Äî value lives in each wrapper's own state.
- *Tree / panel-stack* (batch 8) ‚Äî `TreeDemo` drives `useTreeState(PG_TREE_NODES)` (the hook is
  immutable, so a module-level node constant is safe to reuse across mounts); `PanelStackDemo` holds
  a controlled `stack` seeded with a single root panel whose `renderPanel` pushes a Detail panel ‚Äî
  exercising the real push/pop + back-button. Both inline. See the
  [forms](../.claude/skills/mithril/reference/forms.md),
  [data](../.claude/skills/mithril/reference/data.md), and
  [dates](../.claude/skills/mithril/reference/dates.md) recipes.

### D. Infrastructure ‚Äî playground may not fit (~3)

## Conventions & verification

- **Branch off `main`, build, verify, open a PR, let CI (`verify`) pass, merge.** One batch per
  PR has worked well. (Don't commit straight to `main` ‚Äî that happened once with batch 2.)
- **Verify before merge:** `pnpm build` (typecheck + vite) and `pnpm test` (344 currently) green.
  Then eyeball new playgrounds at `#showcase/<id>` in **both** themes ‚Äî and **hard‚Äëreload** after
  adding configs because of the HMR staleness above. Console should be clean
  (`?theme=dark#showcase/<id>` for the dark pass).
- **Suggested next step:** groups **A** (PR #39), **B** (PR #40), and **C** (Select family PR #41,
  DataTable PR #42, dates/tree/panel-stack batch 8) are all done. The only thing left is group **D**
  (`resize-sensor`/`overflow-list`/`portal`) ‚Äî behavioral/layout utilities that are a poor fit for a
  controls‚Äëdriven single instance; **leaving them gallery‚Äëonly is the recommended outcome**, so the
  playground push is effectively complete unless you want a bespoke demo for one of them.
