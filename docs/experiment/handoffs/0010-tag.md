## Decisions made (and why)

- **`size?: "medium" | "large"` (no "small")** — Blueprint's Tag uses `NonSmallSize` which excludes "small". We follow the same constraint.
- **`onRemove` signature is `(e) => void`** — simplified from Blueprint's `(e, tagProps) => void` since the modern API doesn't need to pass tagProps back through a generic callback.
- **Tag-specific warning tokens** — Blueprint's solid warning bg is `oklch(from orange-3 l+0.19 c h)` ≈ rgb(255,178,97), not exactly `orange-5` (#fbb360 = rgb(251,179,96)). Text is `oklch(from #111418 l+0.05 c h)` ≈ rgb(28,31,36), not pure black. Added dedicated tokens `--tag-solid-warning-bg/text` with the browser-resolved values.
- **Tag minimal-none bg is theme-invariant** — Blueprint uses `oklch(from gray-1 l+0.16 c h / 0.15)` in BOTH light and dark (gray-1 doesn't change between themes). This resolves to `rgba(141,154,174,0.15)` both ways. Do NOT use `--interactive-hover` here — in dark that resolves to white-based rgba.
- **Tag minimal dark intent text tokens** — Blueprint computes `oklch(from intent-rest l+N calc(c-N) h)` for dark minimal intent text, producing values distinctly different from the simple palette tier -5. Added explicit rgb() token values resolved from the Blueprint reference browser: primary=rgb(123,185,255), success=rgb(141,207,164), warning=rgb(253,179,112), danger=rgb(255,139,136).
- **`data-interactive` attribute instead of `:hover` class** — Unlike Blueprint's CSS class approach, we use a `data-interactive` data attribute to gate hover/active color shifts. This avoids polluting the class name space and keeps the Tailwind `data-[interactive=true]:hover:` pattern consistent with the `data-[active=true]:` pattern from Button.
- **Remove button layout** — Uses negative margins (`-mr-1.5/-mr-2`) to tuck into tag padding space, mirroring Blueprint's `.bp6-tag-remove` approach.
- **Round border-radius = 30px** — Blueprint uses `7.5 * bp-surface-spacing = 30px` (not height/2). This creates a pill that's generous for large tags too.

## Gotchas / things to know

- **Tailwind v4 tree-shaking**: All color utilities in TAG_SOLID/TAG_MINIMAL are literal strings — the scanner emits them. Any runtime `var()` in class position would be tree-shaken away.
- **Dark mode `.dark` scope**: The project uses `@custom-variant dark (&:is(.dark *))` — `dark:` variants work on elements that are descendants of `.dark`. Tags in the gallery are inside the `.dark` wrapper div.
- **Blueprint ref gallery data-compare trick**: Blueprint's `<Tag>` doesn't forward data-* to the outer `<span class="bp6-tag">`. Used `useRef`+`useEffect` in a `TaggedTag` wrapper (same pattern as `TaggedSpinner`/`TaggedProgressBar`) to set `data-compare` on the `.bp6-tag` element post-mount.
- **`small-cross` glyph size**: Blueprint uses `SmallCross` (16px standard, 20px for large tags). Our Icon with `size={16}` uses the 16-grid paths; `size={20}` uses 20-grid. Correct.
