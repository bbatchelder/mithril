## Decisions made (and why)

- **`IconProps extends HTMLAttributes<HTMLSpanElement>`** — The outer span receives all HTML
  attributes (including `data-compare`, `aria-*`, event handlers). A separate `svgProps` prop
  forwards SVG-specific attributes to the inner `<svg>`. This mirrors Blueprint's API where
  `htmlProps` goes to the span wrapper. Keeps the harness `[data-compare]` selector landing on
  the measurable element (span with `color` and `height`).

- **`data-icon` on SVG, harness attributes on span** — Blueprint puts `data-icon` on the SVG
  for CSS selector targeting; user-facing attributes land on the span. We match this structure.

- **`min-w-[auto]` on the span** — Tailwind v4's preflight adds `min-width: 0` globally.
  Blueprint's span has `min-width: auto` (browser default for inline-block). Adding
  `min-w-[auto]` restores the default and eliminates the diff.

- **`text-foreground` for `intent="none"`** — Without this, in dark mode the span inherits the
  body's `color: var(--foreground)` which resolves from `:root` (light foreground #1c2127),
  not from the `.dark` scope. This is the same issue as Card; the fix is identical: explicitly
  reference the semantic `text-foreground` token so Tailwind emits `color: var(--color-foreground)`
  which re-evaluates inside `.dark`. This makes dark-mode icons show `#f6f7f9` (light-gray-5)
  matching Blueprint's dark foreground.

- **Intent colors: palette tiers, not `--intent-*-text` tokens** — Blueprint's icon intent
  colors follow `$pt-intent-text-colors`: blue-2/green-2/orange-2/red-2 in light, and
  `$pt-dark-intent-text-colors`: blue-5/green-5/orange-5/red-5 in dark. Our `--intent-*-text`
  tokens use `color-mix()` for dark (calibrated for outlined buttons) which gives different
  values. Icon component uses `text-blue-2 dark:text-blue-5` etc. directly. **Do not "unify"
  icon and button intent colors** — they use different palette tiers by Blueprint design.

- **`fillRule="evenodd"` on all paths** — Blueprint generates paths with `fill-rule="evenodd"`.
  All vendored paths use this. SVG paths that rely on evenodd for correct inner cutouts (e.g.
  `add`, `cog`, `search`) require this to render correctly.

- **Size rule mirrors Blueprint exactly**: `size >= 20` → 20-grid + `0 0 20 20` viewBox;
  `size < 20` → 16-grid + `0 0 16 16` viewBox. Arbitrary intermediate sizes (e.g. 18) use the
  16-grid paths, which is Blueprint's behavior.

- **No `forwardRef<SVGSVGElement>`** — `ref` forwards to the `<span>` (the outermost element),
  consistent with Blueprint. If SVG ref access is needed, use `svgProps.ref` (not implemented;
  add if needed by a consumer).

- **Vendored subset only (16 glyphs)** — 707 Blueprint icons are not vendored; a curated set
  covers the gallery + what Phase 1–2 components will likely need. See the HOW TO ADD section.

## HOW TO ADD A GLYPH

1. Find the icon name at https://blueprintjs.com/docs/#icons.
2. Open the two Blueprint generated path files:
   - `tools/blueprint-reference/node_modules/@blueprintjs/icons/src/generated/16px/paths/<name>.ts`
   - `tools/blueprint-reference/node_modules/@blueprintjs/icons/src/generated/20px/paths/<name>.ts`
3. Copy the `d` string(s) from each file's `export default [...]`.
4. Add an entry to `ICON_GLYPHS` in `src/components/ui/icons/index.ts` following the pattern.
5. TypeScript's `IconName = keyof typeof ICON_GLYPHS` updates automatically.

## Gotchas / things to know

- **`data-compare` goes on the span wrapper, not the SVG** — the harness queries
  `document.querySelectorAll('[data-compare]')` and measures `color` and `height` on that
  element. Blueprint's span has `height` = icon size and `color` = intent color (or inherited).
  Putting `data-compare` on the SVG would skip the wrapper's computed styles.

- **Dark icon color is NOT `$pt-icon-color` in practice** — Blueprint's `$pt-icon-color = $gray1`
  is documented as the "muted" icon color for use in menus/trees. Standalone icons in normal
  text flow inherit from the text context (foreground). The comparison confirms this:
  Blueprint's `icon-intent-none` in light = `rgb(28,33,39)` (foreground), not `rgb(95,107,124)`
  (gray-1/foreground-muted).

- **`fill-rule="evenodd"` is required** — Without it, some paths with inner rings (add, tick-circle,
  etc.) fill incorrectly (the "hole" fills solid instead of transparent).

- **Tailwind v4 preflight sets `min-width: 0` globally** — any inline-block element without an
  explicit min-width gets `0px` instead of `auto`. We override with `min-w-[auto]`.
